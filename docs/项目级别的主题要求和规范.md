一个真正好用、专业、高密度、高质量、高质感的后台系统，不能只定颜色、圆角和按钮样式，而要建立一套完整的：

产品设计语言 + 页面结构规则 + 组件行为规则 + 工程实施规则 + 质量验收标准

对于你的 Vue3 + Arco Design + VXE-Table 货代后台，我建议把项目整体规范分成 三个层级、十个规范模块。

一、先确定系统的核心定位

建议把整体设计目标统一定义为：

PESDP 设计定位
Professional：专业、可信，符合企业级货代系统特征
Efficient：高效，减少点击、跳转和重复输入
Structured：结构清晰，复杂信息有明确层级
Dense：高密度，但不拥挤、不混乱
Premium：克制、有质感，不廉价、不花哨

所有页面、组件和交互都要通过这五项检查。

设计决策优先级建议固定为：

业务效率
> 信息清晰
> 操作安全
> 系统一致
> 视觉质感
> 装饰效果

这意味着：

不为了“高级感”牺牲信息密度
不为了“简洁”隐藏高频操作
不为了“统一”忽略业务差异
不为了“炫酷”增加动画和渐变
不把营销型 SaaS 的大留白套到高密度后台
二、整体规范必须分成三个层级
第一层：设计原则

决定系统为什么这样设计。

包括：

系统定位
信息密度原则
操作层级原则
一致性原则
高风险操作原则
国际化原则
响应式原则
第二层：设计变量 Design Tokens

决定系统具体长什么样。

包括：

颜色
字号
行高
间距
圆角
边框
阴影
控件高度
表格密度
层级与浮层
第三层：页面与组件模式

决定具体页面怎么落地。

包括：

列表页
查询区
表格工具栏
表单页
详情页
Modal / Drawer
Tabs
审批流程
空状态
错误状态
权限状态

只有三层同时建立，规范才不会停留在“建议”。

三、建议建立的十个核心规范模块

1. 设计总纲

先定义整个项目不可突破的设计边界。

1.1 核心原则
□ 高密度不等于小字体、小按钮、无间距
□ 高质感不等于渐变、阴影、大圆角
□ 简洁不等于隐藏功能
□ 一致性不等于所有模块完全相同
□ 主操作必须突出，次要操作必须降级
□ 高频信息优先展示，低频信息按需展开
□ 系统状态必须明确，不能只依赖颜色表达
□ 页面优先服务业务流程，而不是视觉展示
1.2 页面设计目标

每一个页面必须回答四个问题：

用户当前在哪里？
当前最重要的信息是什么？
用户下一步最可能做什么？
操作完成后系统如何反馈？
2. Design Tokens 视觉变量规范

这是系统统一的基础，必须先于页面改造。

2.1 控件高度

建议采用三档，不要页面自行定义。

紧凑型：28px
默认型：32px
宽松型：36px

使用规则：

表格行内、紧凑筛选：28px
普通后台页面：32px
登录、引导、低密度表单：36px
禁止同一区域混用多种高度

你的货代系统建议以 32px 默认、28px 紧凑 为主。

2.2 间距体系

建议采用 4px 基准：

4 / 8 / 12 / 16 / 20 / 24 / 32

主要规则：

图标与文字：4px 或 6px
同组表单项：8px
按钮之间：8px
查询项行列间距：8px～12px
模块内部：12px～16px
模块之间：16px～24px
页面边距：16px～20px

高密度不是全部压到 4px，而是同类紧凑、异类分组明确。

2.3 圆角

企业后台应克制：

小控件：4px
按钮、输入框：4px～6px
卡片、弹窗：6px～8px
禁止大面积使用 12px、16px、20px 圆角

建议项目统一：

--radius-control: 4px;
--radius-container: 6px;
--radius-overlay: 8px;
2.4 阴影

阴影只用于浮层和层级变化：

页面卡片原则上不使用重阴影
Modal、Drawer、Dropdown 可以使用阴影
表格固定列可使用极弱阴影
普通容器优先使用边框和背景层级

禁止：

每个卡片都悬浮
蓝紫渐变阴影
大面积柔光效果
为了“高级感”增加无意义阴影
2.5 色彩职责

颜色必须按职责划分：

Primary：主操作、当前选中、关键链接
Neutral：普通文本、边框、背景
Danger：删除、作废、不可逆风险
Warning：异常、待处理、风险提示
Success：完成、通过、正常状态
Info：辅助说明

关键规则：

□ Primary 不能成为普通装饰色
□ Success / Warning 不作为普通常驻按钮
□ Danger 不得用于非危险强调
□ 禁止仅依赖红、绿表达状态
□ 同一语义必须使用同一种颜色
3. 页面布局规范

项目必须有统一的页面骨架。

3.1 列表页面标准结构

建议统一为：

页面标题 / 面包屑
查询区
列表工具栏
表格区
分页与统计

不要每个业务组自行组合。

3.2 页面边距

建议：

页面左右：16px
页面上下：12px～16px
模块间距：12px～16px

高密度系统不建议使用 24px～32px 的大面积页面边距。

3.3 内容宽度

货代后台以数据表格为主：

默认使用全宽布局
不设置营销型固定内容宽度
详情页可使用分栏
表单正文可限制阅读宽度，但外围保持全宽结构
3.4 层级结构

建议页面最多维持四级视觉层级：

页面级
模块级
分组级
字段级

不要通过过多卡片嵌套表达层级。

1. 查询区规范

查询区是你项目最重要的统一模块之一。

4.1 查询项分级

建议分成：

固定查询项
展开查询项
高级查询项
固定查询项
高频
用户每次查询都可能使用
通常保持 1～3 行
展开查询项
中频
点击“展开”后在原区域展示
不打断用户当前上下文
高级查询项
条件复杂
组合关系复杂
使用 Drawer 或 Modal
支持保存查询方案
4.2 查询操作层级

建议：

查询：Primary
重置：Text
展开/收起：Text
保存方案：Secondary 或 Dropdown
更多查询：Secondary

查询区只能有一个 Primary。

4.3 查询行为

统一规定：

Enter 是否触发查询
修改条件后是否自动查询
重置是否立即查询
日期默认值
下拉是否远程搜索
多选如何折叠
条件是否进入 URL
是否记忆用户上一次条件

这些属于交互规范，不能让各页面自行决定。

1. 列表和表格规范

VXE-Table 是整个系统的信息核心，必须单独建立规范。

5.1 表格密度

建议定义三档：

Compact：行高 32px
Default：行高 36px
Comfortable：行高 40px

货代系统建议：

默认 36px
高密度工作台 32px
不建议普通列表超过 40px
5.2 表头

建议：

表头高度 36px～40px
表头字重 500 或 600
表头背景只使用轻微中性色
不使用大面积深色表头
表头和数据行必须有明确区分
5.3 边框

建议：

默认保留横向分隔线
竖向分隔线按业务需要启用
财务、报价、复杂数字表格可以使用弱竖线
普通列表避免全网格造成 Excel 感
固定列边界必须明确
5.4 单元格内容

统一规定：

文本溢出策略
Tooltip 展示规则
数字对齐
金额精度
日期格式
状态展示
空值展示
可复制字段
链接字段
敏感字段脱敏

建议：

普通文本：左对齐
数字、金额：右对齐
状态、短码：居中或左对齐
空值：—
禁止混用 --、无、空字符串、N/A
5.5 行内操作

建议：

默认可见操作不超过 2～3 个
高频动作直接可见
低频动作进入“更多”
危险动作与普通操作隔离
Icon-only 必须有 Tooltip 和 aria-label
点击热区默认不小于 28×28px
极限高密度最低 24×24px
5.6 分页

你项目可以明确统一为：

分页位于表格右上方或工具栏右侧
表格底部保留总数和选中数量
所有列表统一每页数量选项
翻页后保持或清空选择必须统一
分页加载时禁止整个页面闪烁

如果采用“右上分页”，必须在整个系统中保持一致，而不是部分页面右上、部分页面底部。

1. 按钮和操作层级规范

建议建立五级操作层级：

一级：Primary

当前作用域最重要的下一步操作。

例如：

查询
保存
提交
新建订单
确认选择

同一视觉作用域最多一个。

二级：Secondary / Outline

重要但不是主操作。

例如：

导出
导入
保存方案
批量设置
模块内添加
三级：Text

轻量、辅助、可逆操作。

例如：

重置
取消
清除
展开
收起
查看规则
四级：Icon-only

高频、空间有限、语义明确。

例如：

刷新
复制
编辑
预览
固定
展开行
五级：Dropdown

低频、次要、危险或高级操作。

但不能把所有危险动作机械地放入 Dropdown。核心危险动作仍可以直接显示 Danger Text。

1. 表单规范
7.1 表单布局

建议明确四种模式：

单列编辑表单
双列业务表单
多列高密度录入表单
表格型批量录入

不要所有页面统一使用同一种列数。

7.2 标签位置

建议：

普通编辑页：顶部标签
高密度横向表单：左侧标签
同一个业务区域必须统一
禁止在同一表单中随意混用
7.3 必填和校验

统一规定：

必填标记方式
失焦校验还是提交校验
错误信息位置
跨字段校验方式
服务端错误如何映射
未保存离开提醒
草稿机制
自动保存机制

错误提示必须说明：

错误发生在哪里
为什么错误
用户应该如何修复

不能只写“格式错误”“操作失败”。

1. Modal、Drawer 和页面跳转规范
Modal

适合：

明确、短流程
确认
少量字段编辑
不需要同时参考大量背景信息
Drawer

适合：

中等复杂编辑
查看详情
边看列表边操作
高级查询
快速处理任务
独立页面

适合：

长流程
多步骤
多模块联动
需要完整 URL
需要长期编辑
复杂订单详情

建议建立明确决策规则，避免开发人员凭感觉选择。

1. 状态和反馈规范

专业系统的质感很大程度来自状态处理。

必须统一：

Loading
Empty
Error
Disabled
Readonly
Success
Partial success
No permission
Offline
Data expired
Unsaved changes
9.1 加载
局部操作使用局部 Loading
表格加载不遮挡整个页面
提交操作防重复点击
超过一定时间显示加载说明
禁止无原因长时间 Skeleton
9.2 成功反馈
轻量操作：Message
重要操作：Message + 页面状态更新
长任务：任务中心或进度反馈
不要每个操作都弹 Modal
9.3 错误反馈

错误信息必须业务化：

错误：

保存失败

更好：

订单保存失败：目的港代码已失效，请重新选择目的港。
9.4 空状态

空状态至少区分：

初始无数据
查询无结果
无权限
数据加载失败
数据已被删除

不同空状态不能共用一句“暂无数据”。

1. 国际化和多语言规范

你的系统有国际化，这部分必须前置设计。

10.1 文案长度

设计时至少考虑英文可能比中文长 30%～100%。

禁止：

按中文字符数固定按钮宽度
Tab 宽度过窄
表头无法容纳英文
使用图片承载文字
10.2 文案结构
禁止字符串拼接
使用完整语句翻译
日期、数字、金额按 Locale 格式化
时区必须明确
单复数必须支持
按钮避免仅用一个含义模糊的词
10.3 布局要求
按钮优先使用最小宽度，不固定死宽
表头支持 Tooltip
输入框 label 支持多行或合理宽度
动态验证英文、德文等长文本
预留 RTL 能力时避免写死 left/right
四、高质感不是装饰，而是以下六件事

一个系统是否有质感，主要看：

1. 对齐是否准确
标签基线一致
按钮高度一致
图标和文字垂直居中
表格列对齐
页面左右边界一致
2. 层级是否清晰
用户一眼知道哪里最重要
主次操作区分明确
状态和操作不混在一起
不依赖大量颜色制造层级
3. 间距是否稳定

系统中不能到处出现：

7px、11px、13px、18px、22px

必须来自统一 Token。

1. 文案是否专业

不要使用：

点我
搞定
处理一下
暂无哦
操作失败，请重试

应该使用明确的业务语言。

1. 状态是否完整

缺少 Loading、错误、空状态、权限状态，再漂亮也不是高质量系统。

1. 交互是否可预测

用户不应该猜测：

点击后是否立即提交
删除能否恢复
重置是否触发查询
关闭弹窗是否丢失内容
翻页是否保留勾选
五、规范必须同时约束设计和代码

只写 UI 文档不够，还要落到工程层。

建议建立：

src/
├─ design-system/
│  ├─ tokens/
│  │  ├─ color.css
│  │  ├─ spacing.css
│  │  ├─ typography.css
│  │  ├─ radius.css
│  │  └─ elevation.css
│  ├─ components/
│  ├─ patterns/
│  └─ styles/
├─ components/
│  ├─ PageContainer
│  ├─ SearchPanel
│  ├─ TableToolbar
│  ├─ DataTable
│  ├─ StatusTag
│  ├─ RowActions
│  ├─ FormSection
│  └─ EmptyState

不要让业务页面直接重复拼装 Arco 原始组件。

例如，不应该每个页面自己写一套：

查询区
展开按钮
重置按钮
表格工具栏
分页
行内操作
空状态

而应该沉淀为项目级组件和模式。

六、建立“页面模式”，不要只建立组件

组件统一不代表页面统一。

建议至少沉淀以下页面模式：

Pattern 01：标准列表页
Pattern 02：复杂查询列表页
Pattern 03：主子表列表页
Pattern 04：标准详情页
Pattern 05：可编辑详情页
Pattern 06：分步骤业务流程页
Pattern 07：高密度批量录入页
Pattern 08：数据看板页
Pattern 09：Modal 快速编辑
Pattern 10：Drawer 详情处理

每个模式都包含：

页面结构
组件顺序
间距
操作层级
Loading
Empty
Error
响应式
权限状态

这样 AI 才能按“模式”生成，而不是每次重新设计。

七、建立强制质量门禁

规范必须可以验收。

建议每个页面上线前按八个维度评分。

7.1 八维评估

每项 10 分，总分 80 分：

一致性
信息密度
结构层级
视觉质感
交互效率
专业文案
业务流程完整度
工程可交付性

建议上线门槛：

总分不得低于 68 分
任何单项不得低于 7 分
7.2 页面验收清单
□ 页面是否符合标准页面模式
□ 是否存在多个 Primary 争抢注意力
□ 高频动作是否直接可见
□ 低频动作是否合理收纳
□ 查询区是否支持重置、展开和方案
□ 表格密度、对齐、空值是否统一
□ 是否覆盖 Loading / Empty / Error
□ 删除、作废等危险操作是否有保护
□ 多语言文本是否溢出
□ 1366px 笔记本是否可正常使用
□ 键盘、Tooltip、aria-label 是否完整
□ 颜色、间距、圆角是否全部来自 Token
□ 是否重复实现已有公共组件
□ 是否存在写死颜色、尺寸和文案

第三阶段：建立样板页面

选择三类代表页面：

一个标准列表页
一个复杂详情页
一个高密度录入页

将其作为项目基准页面。

第四阶段：分批迁移旧页面

按业务价值和使用频率排序：

高频核心页面
> 对外演示页面
> 问题严重页面
> 普通业务页面
> 低频历史页面

不是先从最简单页面开始，而是先做最能形成标准的页面。

十、最核心的结论

你的项目整体规范不应该只是：

主题色是什么
按钮多高
圆角多少
表格行高多少

而应该形成以下闭环：

设计原则
→ Design Tokens
→ 公共组件
→ 页面模式
→ 业务规则
→ AI执行规范
→ 自动检查
→ 人工评审
→ 质量门禁

最终目标不是让每个页面“看起来差不多”，而是做到：

用户在任何业务模块中，都能用相同的认知方式理解页面、查找信息、执行操作和处理异常。

这才是一个专业、高密度、高质量、高质感系统真正需要的项目级规范
